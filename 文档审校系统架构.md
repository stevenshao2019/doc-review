## 二、三层架构

```
┌─────────────────────────────────────────────────────────────┐
│  接入层（Entry Layer）                                        │
│  /document-review <文件路径>                                   │
│  作用：统一入口、类型路由、参数解析                                │
├─────────────────────────────────────────────────────────────┤
│  引擎层（Engine Layer）                                        │
│  ├── /pdf-review          → 15 Agent PDF学术审校             │
│  ├── ppt-quality-review   → 7 Phase PPT内容审校             │
│  └── lightweight-review   → 3-5 Agent 轻量文本审校           │
│  作用：文档类型专用的审校逻辑，保持独立演进                          │
├─────────────────────────────────────────────────────────────┤
│  输出层（Output Layer）                                        │
│  output/document-review/<文件名>/                              │
│  作用：统一输出结构、统一确认包格式、统一报告模板                     │
└─────────────────────────────────────────────────────────────┘
```

**核心原则：接入层做路由，引擎层做审校，输出层做统一。**

引擎层的每个子模块保持完全独立，可以独立测试、独立升级、独立调用。

## 三、类型路由逻辑

### 3.1 路由规则

```
输入文件路径 → 提取扩展名（小写） → 匹配规则
```

| 扩展名 | 路由目标 | 审校模式 | 说明 |
|--------|---------|---------|------|
| `.pdf` | `/pdf-review` 命令 | 全量审校（15 Agent） | 引用已存在的 pdf-review.md |
| `.ppt` | `ppt-quality-review` Skill | 全量审校（7 Phase） | 引用已存在的 SKILL.md |
| `.pptx` | `ppt-quality-review` Skill | 全量审校（7 Phase） | 同上 |
| `.md` | 轻量审校（内联定义） | 轻量审校（3-5 Agent） | 新增 |
| `.doc` | 轻量审校（内联定义） | 轻量审校（3-5 Agent） | 新增，需先转docx |
| `.docx` | 轻量审校（内联定义） | 轻量审校（3-5 Agent） | 新增 |

### 3.2 未知类型处理

```python
UNSUPPORTED_EXTENSIONS = {'.exe', '.zip', '.tar', '.gz', '.jpg', '.png', '.gif', '.mp3', '.mp4'}
SUPPORTED_EXTENSIONS = {'.pdf', '.ppt', '.pptx', '.md', '.doc', '.docx'}
```

- 未知扩展名 → 报错退出，提示支持的格式
- 文件不存在 → 报错退出
- 空文件 → 报错退出

## 四、输出目录规范

### 4.1 统一输出结构

```
output/document-review/<文件名>/
│
├── pdf/                          ← PDF专用（从原 output/PDF审校/ 复制）
│   ├── 问题报告/                   完整问题报告.md + 项目综合总结报告.md
│   ├── 核查报告/                   引文核查报告.md + 人名书名核查报告.md
│   ├── 修改决策清单.md
│   ├── 修改版/                     审校修改版.pdf
│   └── analysis/                  PDF分析中间数据
│
├── ppt/                          ← PPT专用（从原 output/PPT审校*/ 复制）
│   ├── 问题报告/                   各PPT问题报告
│   ├── 修改报告/                   修改记录 + A+B类修改汇总
│   ├── 修改版/                     审校修改版.pptx
│   └── 归档/                      最终交付清单 + 项目总结
│
├── md/                           ← MD/DOC专用（轻量审校直出）
│   ├── 全文文本.txt
│   └── 分析数据.json
│
├── reports/                      ← 统一跨类型报告
│   ├── 环境分析报告.md
│   ├── 完整问题报告.md
│   ├── 综合总结报告.md
│   └── 文档审校总览.md              跨格式汇总
│
└── human-confirm/                ← 统一人工确认包
    ├── 1-剩余问题总表.md
    ├── 2-B类问题确认.md              radio单选确认
    ├── 3-C类问题确认.md              引文/译名专项确认
    └── 4-D类问题确认.md              checkbox多项确认
```

### 4.2 与旧输出目录的兼容关系

```
旧路径                             新路径
──────────────────────────────────────────────────
output/PDF审校/<文件名>/    →     output/document-review/<文件名>/pdf/
output/PPT审校问题报告/     →     output/document-review/<文件名>/ppt/问题报告/
output/PPT审校修改版/       →     output/document-review/<文件名>/ppt/修改版/
（新增）                   →     output/document-review/<文件名>/md/
（新增）                   →     output/document-review/<文件名>/reports/
（新增）                   →     output/document-review/<文件名>/human-confirm/
```

旧路径保持不变，新路径通过复制/链接引用旧路径的输出。

## 五、统一人工确认体系

### 5.1 分类确认规则

| 类型 | 确认方式 | 说明 |
|------|---------|------|
| A类 | 仅报告，不进入确认 | 确定为错误，直接报告修改建议 |
| B类 | **radio 单选** | 每条提供两个选项：「建议修改」或「保持原文」，必须二选一 |
| C类 | **引文/译名专项确认** | 每条提供「确认采纳」「需作者确认」「保持原文」三选一 |
| D类 | **checkbox 多项确认** | 每条提供多个检查项，每项独立勾选确认 |

### 5.2 B类确认格式（radio单选）

```
### B1-01 第X页

- **问题**：...
- **建议修改**：...
- **修改理由**：...
- **是否建议修改**：[  ] 建议修改   [  ] 保持原文
- **确认意见**：___________________
```

### 5.3 C类确认格式（引文/译名专项确认）

```
### C1-01 第X页

- **类型**：引文 | 译名
- **原文**：...
- **建议修改**：...
- **依据**：...
- **确认结果**：[  ] 确认采纳   [  ] 需作者确认   [  ] 保持原文
- **确认意见**：___________________
```

### 5.4 D类确认格式（checkbox多项）

```
### D1-01 第X页

- **页码**：第X页
- **问题**：...
- **涉及**：引文格式
- **原文**：...
- **需要确认**：
  - [  ] 检查项1：...
  - [  ] 检查项2：...
- **建议**：...
- **确认意见**：___________________
```

### 5.5 Web 8230 确认系统交互

```
/document-review
    │
    ├── 1. 生成确认包 → 写入 human-confirm/
    ├── 2. POST → http://localhost:8230/api/confirm
    │      { project, output_dir, b_confirm, c_confirm, d_confirm }
    ├── 3. 轮询 GET → http://localhost:8230/api/status/<project>
    │      等待 confirmed 或 timeout
    ├── 4. 回写确认结果 → 追加到 human-confirm/ 各文件
    │
    └── 降级策略（8230不可达）
         └── 提示本地编辑确认包，完成后再运行 --apply-confirm
```

## 六、兼容策略

### 6.1 旧文件不动

| 约束 | 策略 |
|------|------|
| `pdf-review.md` | 完全不动，独立可调用 |
| `ppt-quality-review/SKILL.md` | 完全不动，独立可调用 |
| `output/PDF审校/` | 保留，作为旧格式输出 |
| `output/PPT审校*/` | 保留，作为旧格式输出 |

### 6.2 新增文件不影响旧系统

| 新文件 | 影响 |
|--------|------|
| `document-review.md` | 仅新增，不影响旧命令 |
| `文档审校系统架构.md` | 仅新增，纯参考文档 |
| `output/document-review/` | 仅新增独立目录 |

### 6.3 引用方式

- 新系统通过 **读旧命令的指令** 来"调用"旧能力
- 不通过任何脚本、钩子或配置来耦合新旧系统
- 旧系统可直接调用 `/pdf-review`，完全绕过新系统

### 6.4 演进策略

```
阶段一（当前）：PDF审校能力（已验证）
                ↓
阶段二（本次）：统一入口 + PPT/MD/DOC路由
                ↓
阶段三（未来）：批量处理、定时审校、自动报告分发
```

每个阶段向后兼容，不破坏前一阶段的能力。

## 七、系统边界

### 7.1 范围内

- 学术PDF审校（15 Agent）
- PPT内容审校（7 Phase）
- Markdown文档审校（轻量）
- Word文档审校（轻量）
- 统一人工确认（B radio / C 引文译名 / D checkbox）
- Web 8230 确认系统

### 7.2 范围外（未来扩展方向）

- 图片OCR审校
- 扫描件PDF审校
- 批量目录审校
- 自动生成修改版文件
- 版本对比与增量审校
- 定时/CI触发审校
